# 導入

非同期パイプラインの実装における、ワーカー管理、チャネルハンドリング、流量制御といった定型的な並行処理をカプセル化する。これにより、利用者はコアとなるビジネスロジックの実装に集中できる。

# 概要

**`LaunchWorkers`**と**`Loop`**という2つの主要関数を提供する。`LaunchWorkers`がタスク実行を担うワーカーゴルーチン群を管理し、`Loop`がタスクの供給と結果の集約を一元管理する。この協調により、**N:Mの非対称なタスクフロー**を許容し、**バックプレッシャー**によって安定性を保つパイプラインを構築する。

# LaunchWorkers

指定されたタスク関数を実行するワーカーゴルーチン群を起動・管理する。ワーカーは入力チャネルからタスクを受信し、処理結果を出力チャネルへ送信する。

# Loop

パイプラインの制御を担う。`select`文により「タスクのディスパッチ」と「結果の受信」をノンブロッキングで同時に待ち受け、パイプライン全体のタスクフローが滞らないように制御する。

# 動作原理

### N:Mタスクフローの仕組み

本パイプラインは、タスクの入力と出力が1:1の関係に束縛されない。この柔軟性は、主に`Loop`に渡される`update`関数のシグネチャによって実現される。

`update`関数は、1つの処理結果(`Res`)を引数に取り、戻り値としてタスクのスライス(`[]Req`)を返す。この設計が、結果と次のタスクの間に非対称な関係を生むことを可能にする。

-   **1:N**: `update`関数が複数の要素を持つスライスを返せば、1つの結果から複数の新しいタスクが生成される。
-   **1:0**: `update`関数が`nil`または空のスライスを返せば、タスクは生成されず、その系統の処理は終了する。

さらに、パイプラインは汎用的な部品であるため、その前段にアダプタとなるロジックを配置することで、より複雑なフローも構築できる。例えば、1つのイベントを複数のタスクに分岐させる（Fan-out）、あるいは複数の結果を待ってから1つのタスクを生成する（Fan-in）といった処理を組み合わせることで、システム全体としてM:Nのタスクフローが実現される。

### キュー上限とアルゴリズムの安定性

`Loop`は内部にタスクキューを持ち、そのサイズはドメインロジックから提供される`maxQueueSize`によって制限される。この上限は、アルゴリズムが短期的に生成するタスクのバーストを吸収するためのバッファとして機能するため、探索アルゴリズムがそのような挙動をすることが予期される場合、開発者はこの値を適切に設定する必要がある。

タスクが継続的に`maxQueueSize`を超えて滞留する状況は、システムの一時的な高負荷や流量の問題ではなく、アルゴリズムの設計に本質的な欠陥があることを示唆する。例えば、探索アルゴリズムのようなフィードバックループを持つシステムにおいて、1つのタスク完了が平均して1つ以上の新たなタスクを生成するような、発散的な分岐過程に陥っている状態がこれにあたる。

これは後段の遅延を前段に伝播させて流量を調整する「バックプレッシャー」とは概念が異なり、アルゴリズムが安定性の条件を満たしていないことを示す重大な兆候である。システムはこの種の異常を検知しエラーとみなして終了する。

# パイプラインの安定性

本パッケージはタスクフローを制御するが、パイプライン全体の安定性を保証する責務は利用者のドメインロジックにある。安定性は、生成されるタスクの総数がシステムの処理能力を超えないことであり、主に2つのアプローチで達成される。

1.  **長期安定性**: システムが長時間稼働する場合、1つのタスク処理が生成する新しいタスクの平均数が1以下に収束する必要がある。これにより、タスクキューが無限に増大することを防ぐ。
2.  **有界実行**: タスクの生成率が一時的に1を超えるアルゴリズムであっても、ドメインロジックがタスクの総数に明確な上限（例: 探索回数、深さ）を設けることで、キューのバーストを許容範囲内に抑え込み、最終的にパイプラインを停止させる。
